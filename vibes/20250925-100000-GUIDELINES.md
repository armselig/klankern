# Project Guidelines

This document outlines the core principles, technology choices, and development practices for our project.

## 1. Project Idea

We aim to implement a web application for families to manage common family tasks together and collaborate.

### 1.1 Key Features

- Shared to-do-lists (e.g., shopping lists, chores), with tasks assignable to family members.
- Shared calendar for appointments and events, with entries assignable to single or multiple family members (e.g., mom and child1 go to the swimming course on Monday, on Friday dad and child2 need to go to the doctor).
- A board for short notes or photo uploads, similar to post-its on a refrigerator (e.g., "dinner is in the fridge!").
- Progressive Web App (PWA) with offline support.
- AI agent should suggest additional features.
- Authentication and authorization: an admin user with full control, parents with more rights than children, and configurable access.
- No new user registration; admin will manually add users for family privacy/security.
- Strong security measurements.

### 1.2 Database

- PostgreSQL will be used as the primary database.
- It should run within a container with persistent storage.
- For containerization, we will exclusively use Podman.

### 1.3 Communication

Suggestions for frontend and backend communication will be developed as part of the architecture.

## 2. Architecture Principles

Our architecture will strive for:

- **Simplicity:** Keep solutions straightforward and easy to understand.
- **Maintainability:** Design for long-term ease of modification and bug fixing.
- **Readability:** Ensure code is clear, well-documented, and easy for others to comprehend.

## 3. Technology & Tooling

### 3.1 Core Technologies

- **Runtime:** Use the latest Node.js LTS version.
- **Package Manager:** Utilize the latest pnpm, managed via Corepack.
- **Language:** TypeScript will be used throughout the project.

### 3.2 Development Tools

- **Dependency Management:** Avoid unnecessary dependencies. Prefer implementing simple utilities (e.g., debounce, throttle) locally instead of importing large libraries like Lodash.
- **Linting & Formatting:**
    - Choose modern, reliable, and fast toolings (e.g., Rust-based implementations over JavaScript/TypeScript where possible).
    - Use Prettier for formatting and ESLint for linting.
    - Ensure linters are in place for all languages used in the project (e.g., HTML, CSS, TypeScript, Markdown).
- **Git Hooks:** Prefer Lefthook over Husky. Consider implementing custom, simple git hooks if it proves easier.

## 4. Documentation & Version Control

### 4.1 Code Documentation

- Use extensive JSDoc comments for JavaScript and TSDoc for TypeScript to ensure clear code documentation.

### 4.2 Version Control

- **Commit Messages:** Adhere to conventional commit messages.
- **Changelog:** Implement a system to automatically maintain a `./CHANGELOG.md` file, updating with each version change (major, minor, patch).

## 5. AI Agent Collaboration

- **Agent-Relevant Files:** Keep all files relevant for AI agents within the `./vibes/` directory.
- **Interaction Diary:** Proactively and autonomously maintain a time-stamped diary of our interactions in `./vibes/diary.md`. New entries will be added as we proceed.

## 6. Human-Edited Files

- The `vibes/PROJECT.md` file is intended for human edits only and should not be modified by AI agents.
